描述

此活动的用途是：

  • 协调项目团队对于系统的理解。
  • 对收集到的项目干系人请求的结果执行高级分析。
  • 优化远景，以获取体现系统特征的关键功能（请参阅制定远景）。
  • 优化用例模型，以包含概括出的用例（请参阅查找参与者和用例）。
  • 在补充规范中概括非特定于用例的需求（请参阅开发补充规范）。
  • 以更为结构化的方式开始捕获需求引发活动的结果。

侧重于问题分析和了解项目干系人需要的活动创建了关键系统定义的早期迭代，包括远景中定义的功能和详细需求的第一次概括。此活动定义了系统，它侧重于通过概括参与者和用例的内容来更完整地确定参与者和用例（请参阅查找参与者和用例），以及在补充规范中概括非特定于用例的需求（请参阅开发补充规范）。

在确定其他的通用术语时，词汇表也得到优化（请参阅获取通用词汇表）。

在定义新的需求时，记录这些需求的所有依赖性（例如， 可跟踪性）非常重要（请参阅 管理依赖性）。

人员配备

虽然鼓励项目团队的所有成员以团队主人翁精神和奉献精神参与定义系统，但这项工作主要由扮演系统分析人员角色的职员来协调和开展。因为这项工作经常要求在多个需求之间进行权衡，以最有效地利用有限的开发资源，对于开展这项工作的系统分析人员来说，交际、协商和调解是重要的技能。
使用
使用指导信息

该活动主要在先启和精化阶段中的迭代中执行，但在管理范围和响应不断变化的需求以及项目条件的其他变更时，可按照需要重访。

关键注意事项

请注意，要在此能力模式中执行的活动并不按顺序执行。实际上，很多情况下这些活动是同时执行的。 例如，确定参与者和用例（查找参与者和用例）时，我们可能遇到实质上并不与某特殊用例一致的需求，那种情况下需求可能定义在补充规范（制定补充规范）中。 相反，在确定非特定于用例的需求（例如，系统范围的需求）时，我们可能遇到仅适用于某特定用例的需求，在那种情况下，需求与特定用例相关联。
