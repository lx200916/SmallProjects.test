工作产品：软件需求规范
该工件获取整个系统或该系统某一部分的软件需求。
用途

软件需求规范（SRS）侧重于收集和组织与您的项目有关的所有需求。以正式的 IEEE830 样式的文档来收集项目软件需求是十分有用的。

由于您发现可使用几种不同的工具收集这些需求，因此意识到可能要在几种不同的工件和工具中收集需求是很重要的。正因于此，我们将针对某个包中的 SRS 收集需求，这个包可能是单个文档或是描述需求的多个工件的集合。
（请参阅“更多信息”部分来获取其他指导信息）。

SRS 包控制着项目整个开发阶段的系统发展，如对“远景”文档添加或修改新特性（它们将在 SRS 包中详细说明）。

以下人员使用软件需求规范：

  • 设计人员在定义类的职责、操作和属性时，以及在使类适应于实施环境时会使用 SRS 包作为参考。
  • 实施者在实施类时会将 SRS 包作为输入。
  • 项目经理在计划迭代时会将 SRS 包作为输入。
  • 测试人员在考虑需要哪些测试时会将 SRS 包作为输入。

简述

软件需求规范（SRS）记录系统或系统某一部分的完整软件需求。

一个 SRS 可能有许多不同的安排。有关其他指导信息，请查阅定制部分。

定制
说明选项

一个 SRS 可能有许多不同的安排。请查看此页面顶部表中的模板和示例部分，以了解项目环境中相关的安排。有关该工件的进一步精化（包括 SRS 组织的其他选项），请参阅 [IE830]。

软件需求规范应该提供完整的软件需求定义（功能和非功能的）。但是，特定于用例的需求（功能的和非功能的）应该记录在用例中，而系统范围的需求（功能的和非功能的）应该记录在补充规范中。可以将这些单独的工作产品打包在一起来定义软件需求规范（SRS）。
