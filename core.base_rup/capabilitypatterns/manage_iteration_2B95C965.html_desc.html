描述
此活动获取必要的资源来执行迭代（在任务：获取人员和任务：启动迭代中），分配要完成的工组（在任务：启动迭代中），最后在任务：评估迭代中评估迭代的结果。迭代以 任务：迭代验收复审结束，该复审根据迭代评估确定迭代的目标是否实现。

或者，在周期较长的迭代中，项目经理可能认为，通过在迭代的中途展开一次迭代评估条件复审来重新协调管理人员、技术人员、客户和其他项目干系人的期望可能会是比较谨慎的做法。此复审将主要基于 测试计划进行；在此复审中，项目将通过非常具体的方式来展现迭代的计划内容。这使得，如果存在对于迭代计划的目的的误解，将有机会进行“期中修正”。

人员配备

此活动中的各种活动要求项目经理具备多方面的技能；虽然项目经理可以依靠人力资源部门来提供候选项目成员，但最终还是要自己决定人员选择。 面试技能是一项非常宝贵的技能，它是挑选优秀人员的经验累积。项目经理需要具有规划、领导和团队构建能力，这样才能启动迭代 － 适当地分配工作，并在得力的人员团队中实现抽象的迭代计划。

迭代的评估条件必须客观而明确地设定，所以迭代评估要求项目经理善于进行分析，同时具有客观的头脑。

这些复审的管理复审员需要富有应用领域的经验，并能够准确地确定哪个需求很重要，哪个需求可忽略或放松。虽然对于迭代中哪个需求得到实现、哪个需求未能实现，任何人都不应存在任何疑问，但所有需求的重要性是不同的，也不是恒定不变的。在迭代过程中，将会不断地获得新知，而环境也可能会改变。例如，在迭代开始时，可能会有一个评估条件指定某个功能的响应时间必须小于等于 0.25 秒。 假设事实证明这很难实现，并且在进行迭代验收复审之前，项目只能实现 0.5 秒的响应时间。较低的响应时间是能够实现的，但需要花费大量的资源。但是，客户在看到操作环境中的功能演示之后，认为 0.5 秒是完全可以接受的。

迭代仅仅未能实现此指标并不会产生明显的影响。对于项目经理和管理复审员，最好的做法是同意放松此需求，并增强其他方面的能力作为补偿。管理复审员（和项目经理）需要富有经验和信心，作出这种不会危及项目远景的折衷。
