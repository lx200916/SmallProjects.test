描述

在制定软件开发计划的组成和附件之后，需要对整个计划进行正式复审，以评估可行性和项目干系人的接受度。它将成为下一次迭代中制定精细计划的基础（迭代计划）。

创建这些工件的主要工作在先启阶段的早期进行。项目经理还需要比较软件开发计划的所有其他添加项，并在 任务：编写软件开发计划中汇集它们。

人员配备

很明显，此活动强调的是项目经理的估计、规划和写作技能。另外，项目经理还必须确保在制定这些计划时从项目干系人那里获得有关的经验或信息，因而演示和沟通技能也非常重要。管理复审者需要有估计相关业务或技术领域的经验，并且应能对项目经理所作的假设是否有效作出判断。管理复审者还应该非常了解 Rational Unified Process 才能判断软件开发计划中的 开发案例是否正确。

使用
使用指导信息

主要工作在先启阶段的早期进行；此后，在每次迭代开始时进行（请参阅活动：优化开发计划）。

理想情况下，估算应基于组织自身的经验，该经验随后将用于校准估算模型，例如 COCOMO。（请参阅 [BOE81] 获取对原始模型的描述，或转到 http://sunset.usc.edu/research/cocomosuite/index.html 查看最近的工作。）如果项目经理使用模型系数的缺省值从头开始，则使用其他方法来验证估算将很重要。同样重要的是使人员和其他项目干系人在估算是否现实以及是否可达到目标方面达成一致意见。但是，项目经理必须考虑对估算进行反馈的人员的经验。很多经验不足的人员只是在猜测数字，然后错误地估算了非常多的工作量；但也有可能在进行工作量估算时显得很无知，估算的数字很小。项目经理在处理经验不足的人员的估算时必须很小心，并且要准备在必要时给出建议，同时让更有经验的同事来帮助他们。请参阅任务：规划阶段和迭代获取关于估算的更多信息。

在进行项目规划复审前，应该通过内部走查和复审来评估所有附带的计划和软件开发计划的某些部分。
