活动：规划下一次迭代
此活动的目的是创建迭代计划，这是用于指导下一次迭代的精细计划。

用途
此活动的主要目的是创建迭代计划。制定计划之后，可能需要对业务案例进行调整（例如，如果成本发生了变化或投资收益率的计算受到了软件重要功能发布日期变化的影响）。

描述

迭代计划应由客户和其他项目干系人进行复审，如果令人满意，则应通过迭代计划复审进行核准。此复审也使客户看到项目对于客户参与和资源的期望，尤其是迭代计划用于交付工件或部署软件时，从而使客户可以制定适当的计划。在 Rational Unified Process 中，强烈建议您积极管理迭代范围和资源以符合计划的最终日期，也就是说使用限定时间的方法。这意味着在发生安排问题以及问题被更正时迭代计划可能在迭代过程中发生变更。

人员配备

项目经理需要具有规划、风险管理和估计的综合技能并正确评估迭代的技术内容，因为迭代的内容将根据风险、是否需要演示或交付特定工件和能力以及集成所带来的自然规则等考虑因素作出决定。项目经理需要在准备迭代计划过程中与软件设计人员紧密地合作。管理复审者应与项目经理具有相似的技能，并且也需要对应用领域具有很好的理解。

使用
使用指导信息

此活动在每个阶段的每次迭代中发生（除了最终的项目迭代外）。

强烈建议将此活动作为管理迭代范围和资源的方式，以使迭代和项目保持在正常轨道上。

关键注意事项

项目经理应与软件设计人员紧密合作来定义迭代内容。迭代计划的内容应在内部进行评定、预评估和复审，然后才正式成为迭代计划。

  • 评定迭代评估条件表达式的清晰程度
  • 在内部就工件的构建工时和时间计划达成一致
  • 确保迭代的结果是可测试或可证明的；即迭代将有切实的结果
