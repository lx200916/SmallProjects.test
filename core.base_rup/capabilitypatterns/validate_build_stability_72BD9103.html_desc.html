活动：验证工作版本的稳定性
此任务验证工作版本对于要开始的详细测试和评估工作是足够稳定的。

描述

对于每个工作版本，此项工作侧重于以下方面：

  • 对工作版本的稳定性和可测性作出评估
  • 初步了解工作版本中提供的开发工作或确认工作期望
  • 决定将此工作版本接受为适用于（由评估任务引导的）进一步测试，或对照前一个工作版本执行进一步的测试。

此项工作也称为冒烟测试、工作版本验证测试、工作版本回归测试、完整性检查或测试中的验收。此项工作有助于防止测试资源被浪费在无益和无用的测试工作上。

人员配备

该工作主要集中于测试人员和测试分析人员角色。此项工作所需的最重要的技能包括提供及时的结果、完整性以及针对进一步的测试，将合理的判断运用于评估工作版本的有用性。

分配一个测试团队子集来执行此项工作是合适的；在新的工作版本被验证是稳定的之前，其他团队成员会忽略它，而将他们的工作放在针对来自前一个测试周期的工作版本进行附加测试或适当地改进测试资产上。请参阅活动：改进测试资产。

作为对于按阶段分配相关资源的探索，此活动的测试资源使用的典型百分比为：先启 － 00%、精化 － 05%、构造 － 10% 和移交 － 10%。请注意，在先启阶段中通常没有正式的工作版本。

测试自动化工具的完善性和可用性以及使用工具的必要、必备的技能将影响到为此项工作提供资源。使用自动化工具，此项工作的大部分可快速和高效的执行：若不实行自动化，需要的工作量将显著增加。
使用
使用指导信息

此项工作可能对每个工作版本执行一次 － 不过请注意，它通常不测试每个工作版本。一旦确定此工作版本的稳定性合适，就将侧重点转为活动：测试和评估。当确定此工作版本足以不适合执行进一步的测试时，通常对照前一个合适的工作版本重新开始测试和评估工作。

虽然在某种程度上要依赖于开发工作量的大小，但建议您应该计划使此项工作的适当方面自动执行。对于这些“冒烟”测试中的自动化元素，通常让它们在其他“空闲时间”（如在午餐期间或夜间）以无人照看方式运行。

请注意，除了执行自动化的“冒烟”测试，您还应该计划对新的或有极大变更的软件项执行一系列很小的手动测试。
