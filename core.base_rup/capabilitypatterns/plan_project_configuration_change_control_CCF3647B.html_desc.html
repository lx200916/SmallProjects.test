活动：计划项目配置与变更控制
此活动的目的是确立合适的计划，以管理和控制作为软件开发流程的工作产品而开发的那些工件的变更。

描述

活动侧重于：

  • 确立项目配置管理策略
  • 确立控制产品变更的策略和流程
  • 将这些信息记录在配置管理计划（包括在软件开发计划内）中

CM 策略指的是能够确定、保护和报告已获准在项目中使用的工件。简化和启用标识的方法是：使用适当的工具来控制项目工件，并随着时间对那些工件进行系统标注来确定它们在给定时间点的相对完善性和相互关系。系统标识法是通过归档和基线技术保护项目工件的关键成因。

标准、已记录的变更控制流程确保始终如一地在项目内进行变更，并向适当的项目干系人通知产品的当前状态、所请求的产品变更和这些变更在成本和进度安排等方面的影响。

CM 计划记录将如何计划、实现、控制和组织产品的相关 CM 活动。

人员配备

扮演配置管理员角色的人员本质上需要组织起来，但仍按照项目团队的需要足够灵活地计划配置和变更控制。配置管理员角色支持团队的方式为：确保在项目变更管理工具内反映项目变更策略，使软件开发人员能够轻松地按照既定的开发和批准惯例通过状态变更来转移工件。要求配置管理员角色适当地设置度量，以监视按预期执行 CM 计划且定期进行审计报告，并且该角色与系统管理员角色共同确保 CM 资产的备份得到保护（例如，备份集的现场防火保护，每周非现场存储备份集）。

变更控制管理员是关键的仲裁角色。在这个职位上，软件工作版本中包括的任何给定变更最终取决于项目的变更控制管理员。实际上，通常只准许监视具有重要的潜在影响的那些变更，并且应当根据项目因素（如政治氛围、建立开发人员与客户间信任的需要等等）仔细考虑有关包括或不包括产品变更的任何潜在影响。
使用
使用指导信息

这项工作主要在每个阶段的第一次迭代中执行。虽然这项工作在构造和移交阶段是最重要的，但根据项目惯例，该工作可能在精化甚至先启阶段也很重要。

此项工作不被视为可选的，虽然为了适合项目环境，它将在格式、样式和正式程度上有所不同。还请注意，随着项目周期的推移，变更流程的严格和正式程度倾向于提高，通常在移交变得最为正式和严格。
