活动：管理系统范围
此活动的目的是确保系统的需求清晰明确，并设定迭代的可管理的需求工作集合。
扩展：管理系统范围
描述工作分解结构团队分配工作产品使用
用途
此活动的目的是使所开发系统的范围尽可能明确，并注重于开发周期的可管理需求工作主体。
关系
父代活动
  • 先启阶段迭代 [n]
描述

此活动注重于：

  • 对要包括在当前迭代中的 功能和需求选择的输入信息划分优先级和进行优化（请参阅划分用例优先级）。
  • 为代表重要的核心功能的一个或多个用例定义一组行为场景（请参阅划分用例优先级）。
  • 定义将如何维护可跟踪性，包括要维护哪些 需求属性和 可跟踪性关系（请参阅 管理依赖关系）。
  • 优化远景，以记录解决方案的范围（请参阅制定远景）。

项目的范围由分配给该项目的需求集来确定。按可用资源（时间、人员和资金）管理项目范围是管理成功项目的关键。

系统分析人员角色负责根据适当的项目干系人确定优先级、工时、成本、风险值等需求属性值，这些属性值收集在需求属性存储库中。项目经理角色在计划每次迭代时将使用这些信息，同时这些信息也使软件设计人员角色可以确定对于体系结构有重要意义的场景或完成用例，从而帮助定义体系结构的用例视图（请参阅划分用例优先级）。

人员配备

参与此活动的人员应均为体系结构团队的成员。体系结构团队将促成各个团队成员的对话，以讨论如何最恰当地将需求按优先级排序。 

也可帮助团队负责人培训协商技能，并有助于该项目在组织内以及客户端获得支持。产品／项目支持应具有组织方面的权力，可拒绝超出可用资源的范围变更或是为了适应其他范围而扩展资源。
使用
使用指导信息

应在整个项目中不断执行该活动。

关键注意事项

以 需求属性（如优先级、工时和风险）作为协商需求内容基础，这对于管理范围是一项特别有用的技术。注重属性而不是需求本身，有助于减少协商工作，否则会引起争论。
